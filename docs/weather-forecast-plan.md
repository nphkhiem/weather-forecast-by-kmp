# Part A — Specification

## A.1 Objective

See today + a 7-day forecast (rain %, wind, humidity) for up to 6 city-searched areas,
on Android and iOS from one shared Compose codebase, backed by a SQLDelight cache that
keeps OpenWeatherMap calls well under the 1,000/day free tier.

## A.4 Layer architecture (per `android-api-design`)

```
Composable  →  ViewModel (StateFlow<UiState>, SharedFlow<Event>)
                   ↓ calls
             UseCase (only where logic is reused / non-trivial)
                   ↓
             Repository  ← single source of truth; owns TTL + coalescing
              ↙        ↘
   RemoteDataSource     LocalDataSource
   (Ktor + One Call)    (SQLDelight Flow queries)
```

Rules enforced: three model families kept separate (`Dto` in `data/remote`, DB rows
generated by SQLDelight in `data/local`, **domain models** are what ViewModels see);
mapping lives in `data/mapper`, never in ViewModels; DTOs never cross into presentation;
`StateFlow` for state, `SharedFlow` for one-shot events; every repository method returns
`Flow` (observe) or a `Result`-like type (one-shot); `DispatcherProvider` injected
everywhere I/O happens.

## A.5 Domain models (shape)

```kotlin
data class Location(val id: Long, val name: String, val country: String,
                    val state: String?, val lat: Double, val lon: Double, val sortOrder: Int)

enum class Units { METRIC, IMPERIAL }

data class WeatherCondition(val owmCode: Int, val group: String, val description: String, val icon: WeatherIcon)

data class CurrentConditions(val temp: Double, val feelsLike: Double, val humidity: Int,
                             val windSpeed: Double, val pop: Double, val condition: WeatherCondition)

data class DailyForecast(val date: LocalDate, val tempMin: Double, val tempMax: Double,
                         val humidity: Int, val windSpeed: Double, val pop: Double, val condition: WeatherCondition)

data class HourlyForecast(val time: Instant, val temp: Double, val pop: Double, val windSpeed: Double)

data class Forecast(val location: Location, val current: CurrentConditions,
                    val daily: List<DailyForecast>,   // index 0 = today, 1..7 = next 7
                    val hourly: List<HourlyForecast>,  // next ~48h, used by Detail
                    val units: Units, val fetchedAt: Instant)
```

`pop` = probability of precipitation (0.0–1.0) = the "rain prediction". Error surface:

```kotlin
sealed interface WeatherError {
    data object Network : WeatherError          // no connectivity / timeout
    data object RateLimited : WeatherError      // HTTP 429 — budget hit
    data object Unauthorized : WeatherError     // HTTP 401 — key lacks One Call 3.0
    data object NotFound : WeatherError          // geocoding empty / 404
    data class Unknown(val cause: String) : WeatherError
}
```

## A.6 OpenWeatherMap contract

- **Forecast:** `GET https://api.openweathermap.org/data/3.0/onecall`
  `?lat={lat}&lon={lon}&exclude=minutely,alerts&units={metric|imperial}&appid={key}`
  → `current`, `hourly[48]`, `daily[8]` (index 0 today … 7 = +7). Needed fields per day:
  `dt`, `temp.min/max`, `humidity`, `wind_speed`, `pop`, `weather[0].{id,main,description,icon}`.
- **City search:** `GET https://api.openweathermap.org/geo/1.0/direct?q={city}&limit=5&appid={key}`
  → `[{name, lat, lon, country, state?}]`.
- **Icons:** map `weather[0].id` (OWM condition codes) → bundled local vector via
  `WeatherIconMapper` (offline, no image-loading dependency).
- **Budget:** one forecast call per area refresh; geocoding only during active search.

## A.7 Caching & TTL design (the core requirement)

- **SQLDelight is the single source of truth.** The UI *only* ever observes DB `Flow`s;
  it never reads the network directly.
- **Schema (blob-cache for the forecast, structured for locations):**
  - `SavedLocation(id, name, country, state, lat, lon, sortOrder)` — structured; drives
    the overview list and the "max 6" rule.
  - `ForecastCache(locationId, units, fetchedAtEpochMs, payload)` — `payload` is the
    serialized domain `Forecast` (kotlinx-serialization via a SQLDelight column adapter).
    We always read a forecast whole, so a blob keyed by location is the right size and
    keeps the mapping surface small. (Normalized daily/hourly tables are the documented
    alternative if field-level queries are ever needed, e.g. a widget.)
- **Read path:** `WeatherRepository.observe(locationId)` returns the `ForecastCache`
  `Flow` mapped to domain. On subscribe it evaluates freshness.
- **TTL:** `fetchedAt` vs injected `TimeProvider.now()`. Fresh (`age < TTL`, default
  **30 min**) → **no network call**. Stale + online → fire a refresh that upserts the
  cache → the `Flow` re-emits. Stale + offline → keep serving the cached payload and
  surface a `stale`/`lastUpdated` signal (offline-first).
- **Coalescing:** an in-memory `Map<locationId, Deferred>` (guarded by a `Mutex`) ensures
  concurrent observers of the same area trigger **at most one** in-flight request.
- **Manual refresh:** pull-to-refresh bypasses TTL but still writes through the cache.
- **Units change** invalidates cache for affected areas (units are part of the cache row).

## A.8 Testing strategy (adapting `android-tdd` to KMP — read this)

The `android-tdd` skill targets an **Android-only** JVM stack (JUnit5, MockK, Hilt,
Robolectric, MockWebServer). Most of our code is in `commonMain`, which compiles to
Native too, where those are unavailable. Faithful adaptation:

| android-tdd (JVM) | This project (KMP `commonTest`) | Why |
|---|---|---|
| JUnit5 + `useJUnitPlatform()` | **kotlin-test** (`@Test`, backticked names) | multiplatform test framework |
| MockK | **hand-written Fakes** (skill already prefers fakes for repos) | MockK is JVM-only |
| Turbine | **Turbine** (multiplatform) | unchanged |
| `MainDispatcherRule` (TestWatcher) | `Dispatchers.setMain(StandardTestDispatcher())` in a common helper via `runTest` | coroutines-test is multiplatform; JUnit4/5 rules are not |
| MockWebServer / Retrofit | **Ktor `MockEngine`** | Ktor's multiplatform HTTP test double |
| Room in-memory | **SQLDelight JVM/in-memory driver** in `commonTest` (run on JVM target) | reactive DB tests |
| Hilt test graph | Koin `checkModules` / real modules with fakes | Koin is the DI |

- **given/when/then** naming is mandatory:
  `` `given stale cache and offline, when observed, then emits cached forecast and Stale`() ``.
- **TDD cycle per task:** RED (failing kotlin-test) → GREEN (minimal impl) → REFACTOR.
- **Where tests run:** `./gradlew :composeApp:testDebugUnitTest` (Android JVM) for all
  common logic; `:composeApp:iosSimulatorArm64Test` at least once per phase to prove the
  shared code is genuinely multiplatform (catches Native-only breakage early).
- Optional JVM-only extras (JUnit5 + MockK) may live in `androidUnitTest` if ever needed;
  not required by the MVP.

## A.9 SOLID / Clean-Architecture mapping (explicit, per your requirements)

- **SRP** — one reason to change per class: data source (I/O), repository (policy: cache
  vs network + TTL), use case (one rule), ViewModel (UI state), mapper (translation).
- **OCP / DIP** — presentation and domain depend on **interfaces** (`WeatherRepository`,
  `…RemoteDataSource`, `…LocalDataSource`, `DispatcherProvider`, `TimeProvider`); concrete
  impls are bound in Koin. Swapping OWM for another provider touches only `data/remote`.
- **LSP** — Fakes used in tests are drop-in substitutes for the same interfaces.
- **ISP** — narrow interfaces (`SavedLocationRepository` ≠ `WeatherRepository`); no fat
  "god" repository.
- **Clean boundaries** — domain has zero Android/iOS/Ktor/SQLDelight imports; dependencies
  point inward (presentation → domain ← data).

- [ ] Overview shows every saved area with current temp, icon, today hi/lo and rain %.
- [ ] Detail shows current + 7 daily rows (rain %, wind, humidity) + an hourly rain strip.
- [ ] Opening a fresh area (age < TTL) triggers **no** network call (asserted in a test).
- [ ] Offline with a cached area shows data + a "last updated / stale" indicator, no crash.
- [ ] Every screen renders Loading, Empty (where relevant), and Error states.
- [ ] All shared logic is covered by given/when/then tests passing on **both** JVM and iOS.

# Appendix 2 — Per-task workflow (the loop you repeat)

For **every** task, in order:

1. **Plan** — restate the slice, list files, confirm the acceptance criteria are testable.
2. **RED** — write the given/when/then kotlin-test(s); run them; watch them fail.
3. **GREEN** — simplest implementation that passes (`android-build` Rule 0).
4. **REFACTOR** — clean up; tests stay green and unchanged.
5. **Build both targets** — `:composeApp:assembleDebug` + the phase's iOS test/compile.
6. **Review gate (A.11)** — the full checklist; fix before proceeding.
7. **Commit** — one logical change, revertable.

Only then start the next task. This is the "plan → review → implement → test, then move on"
loop you asked for, applied at task granularity.

# Appendix 3 — Top risks & mitigations

| Risk | Mitigation |
|---|---|
| Key lacks One Call 3.0 (401) | Validate in **T1.5** before building the cache; the account needs the "One Call by Call" subscription. |

| Shared Compose surprises on iOS | Run the iOS simulator at the end of **Phase 3**, not at the very end; `iosSimulatorArm64Test` each phase. |
| SQLDelight blob cache limits future field queries | Documented normalized-schema alternative in A.7; migrate only if a widget/queries need it. |
| `android-tdd` stack assumed (MockK/Hilt/JUnit5) | Adapted to KMP `commonTest` in **A.8** — fakes, kotlin-test, Turbine, Ktor MockEngine, in-memory SQLDelight. |
| Version drift of added libs | Confirm latest patch per A.2 note before wiring in T0.3. |
